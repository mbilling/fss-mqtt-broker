# Delivery — ADR 0042: Durable-plane stress and simulation harness

Decision: [docs/adr/0042-durable-plane-stress-harness.md](../adr/0042-durable-plane-stress-harness.md).

Pre-release area ④ (see ADR 0038's changelog for the four-area plan) — the last of the
four. The durable plane (lease consensus, quorum session-log replication, epoch-fenced
takeover, retained convergence tokens) is tested today by single-fault scenario tests
that synchronize on real time; one already flaked under load. This delivery states the
plane's invariants once as executable checkers, verifies them deterministically over the
pure state machines (many seeds, identical replay — extending ADR 0024's `swim_sim` to
the layer it deferred) and under seed-reproducible fault *schedules* over real in-process
nodes, then gates a bounded profile in CI with a soak profile behind env knobs.

## Plan

| Task | Acceptance criterion |
|------|----------------------|
| **0042-T1** Invariant catalog | The catalog exists as checker code (not prose) covering acked durability, epoch fencing, session singularity, recovery honesty (ADR 0017), retained convergence (ADR 0037), and ADR 0041 bounds; at least one existing scenario test asserts through it, proving the checkers run against real node state. |
| **0042-T2** Pure-core simulation | Seeded schedule generators drive `LeaseMap`, the `cluster_log` fencing/replay logic, and retained token application through reorder/duplicate/drop schedules across ≥1000 seeds each, asserting the catalog after every step; a violating seed panics with the seed and reruns the identical schedule (`REPRO_SEED` knob, `swim_sim` style, no new dependencies). |
| **0042-T3** Whole-cluster stress | One seed composes a fault schedule (kill/restart, partition/heal, frame delay/drop, client churn, takeover storms) and a workload over an in-process multi-node cluster; after quiesce+heal the full catalog plus convergence holds; a failure prints the seed and schedule trace. ## Changelog

- **2026-07-13** — T4 (crash/restart/disk vocabulary) landed: stress nodes run
  over per-node redb dirs; half the seeds restart the killed node over its
  surviving dir mid-schedule (the ADR 0018 crash path under live faults); a
  dedicated test power-cycles the whole cluster and recovers every acked fact
  (which exposed that the harness hub lacked the persistent retained cache
  production always had); `FlakyStore` is a shared fixture injecting terminal
  write errors at the storage seam (acks withheld while on); brownout
  entry/exit runs mid-workload with its refused-but-acked enqueues recorded as
  non-obligations (ADR 0041's documented trade). Two more real finds: zombie
  `dial_forever` tasks outliving their node (a graceful-shutdown bug too —
  fixed with a drop-guard), and the **mesh-whole rule** (seed 4): a gated ack
  may conclude "nobody is owed this" only when every membership-alive peer has
  a live link — under partition it holds, like the durable attach path. Every
  seed now prints its schedule composition. Dev/test builds switch to
  line-tables-only debuginfo (target/ churn 26 GiB → 4.6 GiB).

- **2026-07-13** — T9 (acked means durable, cluster-wide) landed: exhibits ⑤–⑧
  fixed, plus two more found during the fix — ⑨ (SUBACK preceded and ignored the
  durable subscription write) and ⑩, the root cause under most of T3's observed
  losses: durable replication replies routed through the hub command queue
  deadlocked with on-loop appends, so every hub-path durable write failed with
  "no replication quorum" after the 5s RPC timeout on a healthy cluster — invisible
  to the pre-T9 suites because their takeover tests drive the store directly, which
  is exactly the bypass the stress harness refuses. Fixes: replies bypass the hub
  queue (pump → plane); acked cross-node forwards (proto 3) with held obligations,
  sweep retransmission and takeover re-routing; eager inherited-session
  materialization with cluster-wide `ReplicaKeys` discovery, attach-recovery
  gossip, and settle re-delivery for publishes that arrive mid-takeover; the
  retained PUBACK gates on the authority commit; the SUBACK gates on the durable
  subscription write (failure codes + rollback). The harness's waiver
  classifications are deleted — every acked fact is a hard obligation — and
  6-seed soaks hold the catalog with zero losses (and run ~4× faster without the
  deadlock's stalls). Known gaps recorded, not hidden: memory-only in-flight
  QoS 1 redelivery state, fire-and-forget shared-subscription deliveries,
  on-loop store awaits still stalling the hub under partition (follow-up).

- **2026-07-11** — T3 (whole-cluster stress harness) landed — and **found four real
  defects on its first schedules**, all one theme: *acked does not yet mean durable
  cluster-wide*. A seeded 3-node harness over the full production wiring (plus a
  per-node relay for asymmetric bus flaps) runs owner kills, link flaps, and client
  churn against a workload whose obligations ledger records acked facts only,
  classified by the durability path the ack rode; the post-quiesce oracle checks
  recovery honesty, acked delivery, and retained convergence through the T1
  checkers. Exhibits ⑤–⑧ (fire-and-forget cross-node forward; takeover-to-reattach
  enqueue gap; retained ack before authority commit; retained heal liveness) are
  recorded with fix task **T9, ordered before T5**; each is counted loudly per seed
  and anything outside the recorded shapes fails hard. Exhibit ①'s class
  (takeover/resume under load) runs on every seed without flaking — post-quiesce
  assertions are the discipline the original flaky test lacked. CI runs one seed
  (~110s); `MQTTD_STRESS_SEEDS` widens the sweep (4-seed soak verified green).
- **2026-07-10** — T8 (durable leader self-ack) landed: **every exhibit the harness
  program has found is now fixed** — the ledger holds no open defects (exhibit ①,
  the load flake, remains the T3/T5 target). The owner's self-ack counts toward a
  write quorum only once the op is durably applied to the node's own replica copy
  (the same copy its restart-recovery reads), so an acked entry exists on a full
  durable quorum; the T6 re-commit is gated the same way, and truncate/remove keep
  the owner's copy current. Proven by a crash-restart regression through the
  production path. The T2 sim needed no change — it always modeled durable
  self-acks; reality now matches the model the catalog was verified against.
- **2026-07-10** — T7 (epoch-tagged replica entries) landed: **the invariant catalog
  is enforced unwaived across every simulated schedule — no known exhibits remain
  open in the replication glue** (exhibit ④, the volatile leader self-ack, remains
  for T8). Every entry now carries its writing `(epoch, seq)` — replica schema v3
  (fail-closed gate) and peer-bus proto 2 (`ReplicaEntryWire` + `ReplOp::Append`
  reshaped; `PROTO_MIN` raised alongside, legal only pre-release). Replicas keep the
  higher version of an offset (a late duplicate of a superseded attempt never
  overwrites, across reopen too), and the recovery merge resolves same-offset
  conflicts by tag — never read order — and truncates a tail whose tag regresses
  (log matching), which also ends the recovered-retained-token regression. The sim's
  waiver machinery is deleted: the 1000-seed sweeps ARE the strict tests now.
- **2026-07-10** — T6 (takeover re-commit) landed: **acked durability is enforced,
  unwaived, across every simulated schedule again.** Exhibit ② proved to be two
  mechanisms — the adopted-orphan gap (fixed by `ClusterLog::recommit_key`: the
  recovered base is re-replicated to a write quorum at the new owner's epoch before
  the key is served, or the owner refuses with `NoQuorum`) and an offset-space reset
  below a durable truncation watermark (surfaced the instant the waiver lifted:
  an empty merge of a fully-drained queue restarted offsets at 1; recovery now
  threads the reads' truncation floor and continues above it). Three regression
  tests pin both mechanisms end-to-end through the production `GroupRoutedLog`
  path. En route, a **fourth defect** was found by inspection and recorded as
  exhibit ④ with fix task T8: the owner's quorum self-ack is volatile (its own
  durable replica copy is never written by its own appends). Exhibit ③ remains the
  only waived class.
- **2026-07-10** — T2 (pure-core simulation) landed — and **found two real durability
  defects on its first sweep**: seeded schedules (per-delivery fates: counted accept,
  ack-lost, drop, stale defer; offset reuse; takeover after takeover with seeded read
  quorums and orders) over the real `ReplicaState`/`merge_replica_logs`/`LeaseMap`/
  `Placement`, 1000 seeds per scenario, the T1 catalog as oracle. Exhibit ②:
  adopted single-replica orphans are built upon without re-replication, and a later
  takeover can gap them out — discarding the acked tail (fix: T6, takeover
  re-commit). Exhibit ③: offset reuse after `NoQuorum` diverges replicas at an acked
  offset and the merge resolves by read order; the same unfenced adoption can regress
  a recovered retained token behind an acked one (fix: T7, epoch-tagged entries +
  log matching). Both are waived by exact class in the sweeps (loudly counted), kept
  red in an `#[ignore]`d no-waiver test, and recorded in the exhibit ledger; T6/T7
  were added to the plan and run **before T3**. Lease replay determinism and HRW
  placement agreement/minimal-disruption swept clean. Two oracle refinements fell out:
  truncation-issue discharges durability while only truncation-quorum floors
  resurrection, and fencing is per-replica.
- **2026-07-10** — T1 (invariant catalog) landed: the durable plane's guarantees now
  exist once, as executable checkers (`mqtt_cluster::invariants`) — acked durability
  verified against recovery merges, epoch fencing over every replica decision, lease
  epoch monotonicity + map agreement, retained token monotonicity + cross-node
  convergence, session singularity, recovery honesty, bounded structures. Each checker
  is self-tested to *catch* its violation shapes, and three real scenario tests now
  close through the catalog — including the exhibit-ledger takeover test, whose token
  history is now checked end-to-end rather than at two scripted points.
