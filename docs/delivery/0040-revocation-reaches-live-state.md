Workspace green (768 tests), clippy zero warnings.
  - id: 0040-T2
    title: Identity sweep — a successful reload disconnects live clients whose cert serial is CRL'd, whose password user was removed, or whose principal the new connect-ACL denies; untouched sessions keep flowing
    status: done
    date: 2026-07-05
    evidence: "Reloader::attach_identity_sweep(hub, client_crl_build): after every SUCCESSFUL swap the reloader hands the hub a SweepPolicy carrying exactly the Arcs it just published (authorizer + authenticator), the client-listener CRL's revoked serials, the trigger, and the audit sink; a rejected reload sweeps nothing, and a bad client CRL rejects the WHOLE reload (validate-before-swap — same file the TLS verifier enforces per handshake, parsed via the new RevocationList::from_bytes_unverified, PEM or DER). Hub::sweep_identities re-derives each online session's admission verdict against the new policy and evicts on three identity-level facts: presented cert serial on the CRL (cert-revoked), password-admitted subject gone from the credential store (user-removed — via the new Authenticator::password_subject_exists probe, default true, overridden by PasswordAuthenticator and ChainAuthenticator; an existence check on server-held facts, never a credential replay), or connect-ACL deny (connect-denied — Admission now carries the full Identity with groups so sweep-time authz sees exactly what admission-time saw). Every eviction is audited (security.evict, reason + trigger). An unchanged policy evicts no one. Wired in main.rs (MQTTD_TLS_CRL mirrored into the sweep). Tests: hub unit (one sweep evicts all three classes while an untouched session keeps flowing, each audited), reload unit (sweep dispatch on success only; bad client CRL rejects the reload), and three end-to-end acceptance tests, one per revocation class, each proving eviction with NO client action while a bystander keeps flowing: tls.rs a_crl_reload_evicts_the_live_session_of_a_revoked_cert (live mTLS session, CRL published mid-session), reload_auth removing_a_password_user_evicts_their_live_session, reload_acl a_connect_acl_tightening_evicts_the_live_session (opt-in connect rule). Workspace green (773 tests), clippy zero warnings.

    title: Grant sweep — a tightened subscribe-ACL removes matching grants from live routing and durable subscription sets (online and offline sessions) and stops queued replay; no disconnect for permission-only changes
    status: done
    date: 2026-07-05
    evidence: "Two enforcement points, one rule (permission tightened -> grant removed, session kept). ONLINE (Hub::sweep_grants, runs after the T2 identity sweep on the same SweepPolicy): every surviving online session's granted filters are re-authorized under the identity the session was admitted with, checking the same raw filter string SUBSCRIBE-time authorized ($share prefix included) so sweep-time and admission-time verdicts align; denied grants are removed via the existing unsubscribe path (routing table, shared groups, durable set, gossip) and audited (security.evict, grant-revoked, with the filters and trigger). The client is NOT disconnected; its next SUBSCRIBE re-attempt is denied at the admission check (0x80). OFFLINE (resume-time, finish_attach): the hub holds the live authorizer watch (HubCommand::AttachAuthorizer, sent by main.rs before any listener accepts — the same channel connections read, so hub and admission path always see the same policy); on resume every restored subscription is re-authorized under the RESUMING principal's full identity (fresh from authentication, groups included — why this beats a store-wide sweep with subject-only identities), revoked grants are pruned from live routing after the reconcile (ordering matters: pruning before it would persist an empty set on a fresh hub) and persisted, and queued messages that only a revoked grant admits are dropped from replay — a topic a surviving grant also matches still replays (shared-aware matching). No authorizer attached = no re-check (harnesses keep today's behavior). Tests: hub unit the_grant_sweep_removes_revoked_subscriptions_without_disconnecting (revoked filter stops delivering, untouched filter + connection keep working, audited) and a_resumed_session_loses_grants_the_current_policy_denies (queued message for the revoked grant not replayed, surviving grant's replayed, routing pruned durably); integration reload_acl a_subscribe_acl_tightening_stops_delivery_to_a_live_subscription (delivery stops, PING still answers, re-SUBSCRIBE 0x80) and an_offline_sessions_revoked_grant_does_not_replay_its_queue_on_resume (QoS1 queue not replayed after sleeping through the reload, re-SUBSCRIBE 0x80). Workspace green (777 tests), clippy zero warnings.

    status: done
    date: 2026-07-05
    evidence: "Three pieces. (1) RELOADABLE PEER TLS (pays the ADR 0032 deferred item): PeerTls's acceptor/connector move behind watch channels, read per accept / per dial (and per proxy relocation dial — ProxyContext carries the watch too); Reloader::attach_peer_tls folds the rebuild into the same atomic validate-before-swap reload (a failing peer-TLS build rejects the WHOLE reload), so a rotated cluster cert/key/CA is served on the next peer handshake; fixed_acceptor/fixed_connector helpers keep non-reloading setups working. (2) HANDSHAKE GATE: admit_peer_link checks every freshly-handshaken link's remote leaf serial against the LIVE cluster-CRL slot (the same slot the gossip verifier reads per datagram, ADR 0022 T7) on BOTH sides — accept (a revoked node dialing us) and dial (us dialing a revoked node: the TLS verifier alone checks chain + name, not revocation) — closing the re-establishment hole; the dialer keeps redialing so an un-revocation heals. (3) LINK SWEEP: links register their remote serial with the hub (PeerConnected.cert_serial -> Peer entry); SweepPolicy gains peer_revoked (the freshly-swapped cluster CRL) and Hub::sweep_peers tears down revoked links after the T2/T3 sweeps — removing the entry drops the outbound sender, ending the pump and closing the socket; audited (security.evict, peer-revoked). # Delivery — ADR 0040: Revocation reaches live state

Decision: [docs/adr/0040-revocation-reaches-live-state.md](../adr/0040-revocation-reaches-live-state.md).

Pre-release area ② (see ADR 0038's changelog for the four-area plan). Every revocation
mechanism today enforces at admission time; this delivery makes a successful policy
reload sweep **live** state — open client sessions, existing subscription grants,
established peer links — with the two-tier rule: identity revoked → session terminated,
permission tightened → grant removed.

## Plan

| Task | Acceptance criterion |
|------|----------------------|
| **0040-T1** Metadata + eviction | Connections retain principal source (anonymous/password/JWT/mTLS) and leaf serial when a client cert was presented; a hub command terminates a named session (v5 reason `0x87`; v3.1.1 close) reusing the takeover close path; no behavior change yet (mechanism only). |
| **0040-T2** Identity sweep | With a live connected client: publishing a CRL revoking its cert, deleting its password user, or denying its principal in the connect-ACL — each followed by a reload — disconnects it **without any client action**, while a second untouched client keeps flowing. Each eviction audited with its reason. |
| **0040-T3** Grant sweep | A subscriber receiving matching messages stops receiving them after a reload tightens its subscribe-ACL — subscription removed from routing **and** from the durable session set (proven for an offline session too: no replay of the revoked filter's queue on resume); the client is not disconnected; its next SUBSCRIBE is denied. |
| **0040-T4** Peer-bus revocation | Peer acceptor/connector rebuilt on reload (rotated cluster cert served on next peer handshake); a cluster-CRL reload tears down an established link to the revoked node in a two-node cluster; the revoked node cannot re-handshake; the healthy node's clients are undisturbed. |
| **0040-T5** Audit/metrics + closure | `security.evict` events carry kind + trigger; `revocation_evictions_total{kind}` registered; reload audit includes sweep counts; a test pins that a removed user cannot resume their durable session (admission-side); README documents the operational semantics. |

## Changelog

- **2026-07-05** — T4 (peer-bus revocation) landed: the last plane. Peer-bus TLS is
  now reloadable (the ADR 0032 deferred item, paid), every fresh peer handshake is
  gated on the live cluster CRL in both directions, established links record their
  remote serial, and a CRL reload tears revoked links down immediately — proven on a
  routing two-node mesh. All three live-state planes (client sessions, subscription
  grants, peer links) now enforce revocation at reload time; T5 closes with
  metrics/audit polish and docs.
- **2026-07-05** — T3 (grant sweep) landed: tightened subscribe-ACLs now reach existing
  subscriptions — online grants are swept on reload (removed from routing + durable set,
  audited, no disconnect) and offline sessions are re-checked at resume under the
  resuming principal's full identity, with queued messages that only a revoked grant
  admits dropped from replay. The two-tier rule is complete on the client plane:
  identity revoked → session terminated (T2); permission tightened → grant removed,
  flow stopped, session kept (T3).
- **2026-07-05** — T2 (identity sweep) landed: revocation now reaches live client
  sessions. A successful reload hands the hub the exact policy it published; the hub
  re-derives every online session's admission verdict and evicts on cert-revoked /
  user-removed / connect-denied, audited per eviction — proven end to end for all
  three classes (a live mTLS session dies when the CRL naming it reloads; a deleted
  password user's session dies; a connect-ACL lockout dies) with bystanders
  undisturbed. The client CRL joins the validate-before-swap set: a malformed CRL
  rejects the whole reload.
- **2026-07-05** — T1 (admission metadata + eviction primitive) landed: connections now
  carry their revocable admission facts (subject, auth method, mTLS leaf serial,
  protocol) into the hub's online table, and the hub can terminate a named session —
  v5 clients are told why (DISCONNECT 0x87), v3.1.1 just closes, wills and session
  retention behave as for any ungraceful end. Mechanism only; the T2/T3 sweeps drive it
  next.
- **2026-07-05** — ADR proposed and delivery opened. Scope fixed by a live-state gap
  survey: (1) revoked client certs leave open TLS sessions untouched (the existing CRL
  test drops the connection *before* reloading); (2) removed password users keep their
  open connections (auth runs once at CONNECT); (3) tightened subscribe-ACLs grandfather
  existing subscriptions (fan-out performs no authz); (4) established peer links survive
  cluster-cert revocation (peer TLS built once at startup; ADR 0032 deferred its reload);
  (5) gossip-plane revocation (ADR 0022 T7) is already per-datagram — the model to match;
  (6) durable sessions of removed identities stay inert until expiry (kept deliberately —
  see ADR §5); (7) no audit/metric distinguishes revocation reaching live state. Ordering:
  T1 (mechanism) → T2/T3 (client-plane sweeps, independently landable) → T4 (peer plane)
  → T5 (closure).
