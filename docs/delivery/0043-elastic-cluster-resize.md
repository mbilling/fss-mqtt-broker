  - id: 0043-P1
    title: Catch-up replicas — a node entering a group's replica set back-fills the group's log (ReplicaKeys discovery + recover_key quorum reads) behind a durable caught-up watermark, and counts toward NO quorum (append or recovery) until caught up; growing 1→N back-fills laptop-mode single-replica history as the same rule
    status: done
    date: 2026-07-13
    evidence: "Recovery reads now carry a COMPLETENESS verdict with two halves: per-key gap-freedom (ReplicaState::complete — stored offsets contiguous above the truncation low-water, so a joiner holding new appends above a hole it never received reads incomplete) AND the durable caught-up watermark (new replica_caught_up redb table: per group, the replica set this node last completed catch-up against; a stamp from a different cohort — or no stamp — reads incomplete, so an EMPTY joiner cannot claim a key has no history; additive table, no schema bump, survives restart so a crash mid-catch-up resumes rather than fakes completion). recover_key requires a quorum PLUS at least one complete anchor, and the OWN copy obeys the same rule — an unstamped owner cannot fabricate an empty log (fails closed with NoQuorum until its boot sweep stamps; ADR 0017 posture). The back-fill: a catch-up sweep in the lease driver (armed at boot and on every placement membership change, CATCH_UP_SWEEP_EVERY=5 ticks, budget 24) discovers keys per-peer (ReplicaKeys, attributed — a group stamps only once every other set member was heard), heals hollow keys by asking the group owner to recommit_key them (new ReplicaCatchUp frame; owner-side CatchUpSource seam on the plane, implemented by GroupRoutedLog: route → recover → re-commit at the owner epoch, idempotent and fenced, plus a truncation-floor fan so drained prefixes read as acked-away rather than as gaps), and re-stamps pure shrinks of a known cohort immediately (custody unchanged — no data moved on 5→3). Cached group logs rebuild on REPLICA-SET changes, not only epoch changes, so appends and re-commits fan out to joiners. Wire: proto 3→4; ReplicaRead2/ReplicaReadReply2 (request version selects reply shape; pre-proto-4 replies are conservatively incomplete) and ReplicaCatchUp appended at the enum end (ADR 0038 compat). Tests — unit: completeness shapes (gap/hole/truncated-floor), stamp currency + reopen persistence, unstamped-owner-cannot-fabricate, recovery-refuses-a-quorum-of-hollow-replicas (then serves once a complete anchor joins), growing_a_one_node_group_back_fills_the_joiners (store seam), proto4 wire round-trip + catch-up routing (plane seam). Integration (cluster_stress): growing_one_node_to_three_back_fills_and_survives_the_founder — one durable node accumulates acked facts (offline durable session owed 3 acked QoS1 payloads + acked retained), grows 1→3, both joiners reach the caught-up watermark on q/, m/ and r/ keys, the FOUNDER IS KILLED (the only pre-grow copy), and the session resumes present=true on the survivors with every acked payload replayed and the retained value served — zero loss, zero waivers (~17s). Workspace green (all suites), clippy zero warnings."
  - id: 0043-P2
    title: Eager migration on ring change — membership growth triggers the takeover-scan materialization (sessions, retained, interest) for groups whose owner moved, instead of first-touch; the settle/re-route + mesh-whole ack machinery holds acks honest during the window
    status: done
    date: 2026-07-13
    evidence: "The hub now watches the placement member set on its sweep tick (new known_members snapshot): ANY change — growth above all; shrink arrives via SWIM eviction too, overlapping the PeerDead arm — re-arms the takeover window (takeover_reconcile_ticks=8), so every node runs the eager inherited-session scan for the next ~8s and publish acks registered in the window gate on awaiting_settle exactly as after a death (settle releases only when the window is over AND the mesh is whole — the T9/T4 honesty rules, inherited unchanged). Two sides of the move: the NEW owner's scan materializes the moved groups' sessions (all_sessions unions local + remote keys; per-key recovery rides P1's complete-anchor gate, with the old owner — alive and stamped — as the anchor) and gossips their filters, re-targeting held forwards; and the OLD owner now has the missing mirror — release_moved_sessions, run after every scan: offline, not-attaching sessions whose group this node no longer owns drop their LOCAL routing state (durable data stays; the node usually remains a replica) and interest re-gossips without them, so forwards stop targeting a node that can only answer NotOwner (every such ack was being withheld — the pre-P2 state poisoned post-grow publishes indefinitely). Durable-cluster-gated: with local-only session storage the data cannot follow ownership, so routing is kept. Tests — unit: growth_releases_moved_sessions_on_the_old_owner (paused-clock hub with a real idle plane: boot scan gossips the materialized filter, placement growth re-gossips without it). Integration (cluster_stress): growth_migrates_moved_sessions_eagerly_and_acks_stay_honest — offline durable session whose owner MOVES on a 1→3 grow (no deaths): 2 payloads acked pre-grow, 3 acked post-grow against the old owner with the session touched by no client; resume on the NEW owner is present=true and replays all 5 (~12s). Validated non-vacuous: the same test against the pre-P2 hub FAILS (post-grow acks withheld behind the old owner's stale NotOwner routing). Workspace green, clippy zero warnings."

    title: Decommission — an explicit drain (stop new sessions → successors caught up + replica counts restored among remaining members → voter demotion → ADR 0019 leave), observable via the health endpoint, interruptible (crash mid-drain = crash); operator trigger chosen here (signal vs admin endpoint)
    status: done
    date: 2026-07-13
    evidence: "Operator trigger: SIGUSR1 (chosen over an admin endpoint — the health listener is deliberately read-only and unauthenticated; signals match the existing SIGHUP-reload / SIGTERM-shutdown operational grammar). On SIGUSR1, readiness fails immediately (orchestrators steer new traffic away — the ADR 0019 §4 posture standing in for 'stop accepting new sessions' pre-proxy) and the DECOMMISSION DRAIN runs (new mqtt_cluster::decommission::Drain, assembled from the plane's shared handles via DurablePlane::decommission_drain). The drain cannot push data itself — a non-owner writing old-epoch entries would be fenced, and minting epochs is the owner's privilege — so each round it VERIFIES every key it stores against every member of the key's post-departure replica set (Placement::group_replica_set_without — HRW monotonicity keeps every surviving current member in the set) using plain recovery reads, with CONTENT SUFFICIENCY as the verdict: the successor's truncation watermark has reached ours (an acked-away prefix can never be resurrected from a merge that no longer includes us) and every entry we hold is either below that watermark or present in its log. Each shortfall asks the key's group OWNER for a targeted re-commit — new proto-5 ReplicaCatchUpTo frame (appended at the enum end, PROTO_MAX 4→5), served through the CatchUpSource seam by ClusterLog::recommit_key_to: every committed entry re-tagged at the owner's epoch (the P1 recommit convention, idempotent) plus the truncation floor, to ONE node, no quorum gate — the drain re-verifies and re-asks until nothing falls short. Rounds repeat 1s apart; the drain never gives up on its own (an unreachable successor keeps it honestly pending; the operator can always escalate to plain SIGTERM = crash semantics — the drain mutates nothing locally, so interruption at ANY point is just a crash). On completion the ordinary ADR 0019 graceful shutdown runs: SWIM leave → placement eviction → voter demotion via the ADR 0021 reconciler → P2's eager migration materializes the moved sessions on the survivors. Observability: /readyz gains decommission{pending,rounds,complete} while a drain is active (DrainStatus published into the health state). Tests — unit: a_solo_drain_completes_immediately (nowhere to hand off = shutdown); the_drain_hands_off_and_verifies_a_held_key (two-node ring, truncated key, empty successor: round 1 finds the shortfall, targeted re-commit lands entries AND watermark, rounds converge to 0); plane wire test extended (ReplicaCatchUpTo round-trips to the owner's CatchUpSource with its target). Integration (cluster_stress): a_decommissioned_nodes_departure_loses_nothing — FOUR-node cluster (4 members, R=3: removing one adds exactly one NEWCOMER to every set it was in, so the targeted hand-off is load-bearing), acked payloads + acked retained on sets including the leaver, the durable subscriber OWNED by the leaver; drain converges (~120s bound, actual seconds), reports complete, the node dies (kill — harsher than the graceful leave), and everything acked survives: resume present=true on the new owner, all payloads replay, retained serves (~14s). Workspace green, clippy zero warnings."

  it fails against the pre-P2 hub and passes in ~12s with it). Durable-cluster-gated;
  local-only storage keeps routing since its data cannot follow ownership.

- **2026-07-13** — **0043-P1 done: catch-up replicas.** A recovery read now carries a
  **completeness verdict** — gap-free stored offsets above the truncation low-water
  *and* a durable per-group **caught-up stamp** (`replica_caught_up` table: the replica
  set this node last completed catch-up against; survives restart, so a crash
  mid-catch-up resumes rather than fakes). `recover_key` requires a quorum **plus at
  least one complete anchor**, which closes both faces of the hollow-replica hazard: a
  joiner holding new appends above a hole it never received (gap ⇒ incomplete), and an
  **empty** joiner claiming a key has no history (no stamp ⇒ incomplete — an unstamped
  owner cannot fabricate an empty log either). The back-fill itself: a catch-up sweep in
  the lease driver (armed at boot and on every membership change) discovers keys
  per-peer (`ReplicaKeys`), asks each hollow key's owner to `recommit_key` it
  (`ReplicaCatchUp`, proto 4) — the existing fenced idempotent re-commit — plus a
  truncation-floor fan so drained prefixes read as acked-away, not as gaps; groups
  re-stamp when every discovered key is gap-free and every other set member was heard
  (pure shrinks of a known cohort re-stamp immediately — custody unchanged). Cached
  group logs now rebuild on **replica-set changes** too, so appends and re-commits fan
  out to joiners. Wire: proto 3→4, `ReplicaRead2`/`ReplicaReadReply2`/`ReplicaCatchUp`
  appended (pre-proto-4 peers keep the legacy read; their replies are conservatively
  incomplete). Proven end to end by the harness: grow 1→3 under acked facts, wait for
  the joiners' watermarks, kill the founder — session resumes present, every acked
  payload replays, the acked retained value serves (zero waivers).
